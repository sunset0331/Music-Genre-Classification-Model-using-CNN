## 1. Project Overview (Non-Technical)

### What does this project do?
Imagine you hand a computer a music clip and it tells you — *"That's jazz!"* or *"That's classical!"* — without any metadata, lyrics, or artist information. That is exactly what this project achieves.

The **Music Genre Classification System** listens to raw audio files and automatically identifies which genre they belong to. It does this by first converting audio into visual representations (called **mel spectrograms**) and then using a deep learning model (a **Convolutional Neural Network**) that has learned to "see" the difference between genres.

### Why does this matter?
- **Music streaming platforms** (Spotify, Apple Music, YouTube Music) need to auto-tag millions of songs uploaded every day.
- **Music recommendation systems** rely on genre to suggest similar songs.
- **Content creators** and **DJs** need fast genre tagging for large libraries.
- It is a gateway project into **audio intelligence** — a rapidly growing field in AI.

### Who is this built for?
This is an end-to-end machine learning project suitable for:
- Students and developers learning deep learning with audio data.
- Data science portfolio projects.
- A foundation for building music recommendation or audio analysis systems.

## 2. Problem Statement

Given an audio file in `.wav` format, predict which music genre it belongs to from a set of predefined genres.

- **Type of Problem:** Multi-class Classification
- **Input:** Raw audio (`.wav` file)
- **Output:** A genre label (e.g., `blues`, `jazz`, `classical`, etc.)
- **Approach:** Convert audio → image (mel spectrogram) → classify with CNN

## 5. Audio Processing — How Sound Becomes an Image

This is the most important concept in the project. A CNN expects an image as input, not a raw audio waveform. The pipeline solves this by converting audio into a **mel spectrogram**.

### Step-by-Step Audio → Image

#### Step 1: Load Audio
```python
y, sr = librosa.load(audio_path, sr=22050)
```
- `y` is the raw waveform as a NumPy array of amplitude values over time.
- `sr` is the sampling rate (22,050 samples per second).

#### Step 2: Compute Mel Spectrogram
```python
spectrogram = librosa.feature.melspectrogram(y=y, sr=sr)
```
- A **spectrogram** shows how the frequency content of the audio changes over time.
- The **mel scale** re-maps frequencies to match human hearing perception — our ears are more sensitive to differences at low frequencies than high ones.
- The result is a 2D matrix: **frequency bands (y-axis) × time frames (x-axis)**.

#### Step 3: Convert to Log Scale (dB)
```python
log_spectrogram = librosa.power_to_db(spectrogram)
```
- Human perception of loudness is logarithmic.
- Converting power to decibels gives a more perceptually meaningful and visually distinct image.

#### Step 4: Save as PNG Image
```python
plt.figure(figsize=(3, 3))
librosa.display.specshow(log_spectrogram, sr=sr, cmap='magma')
plt.axis('off')
plt.savefig(output_path, bbox_inches='tight', pad_inches=0)
```
- The spectrogram is rendered as a 3×3 inch image with no axes, padding, or labels.
- The `magma` colormap maps intensity to color (black → purple → yellow → white).
- Saved as a `.png` file that the CNN can read like any photograph.

### Why Does This Work?
Different genres have distinct **timbral textures and rhythmic patterns** that show up visually in their spectrograms:
- **Classical** music has smooth, spread-out frequency patterns.
- **Metal** has dense, high-energy distributions across all frequencies.
- **Blues/Jazz** show characteristic melodic contours.
- **Disco/Pop** reveal strong, repeating rhythmic beats.

## 7. Model Design (CNN)

The model is a **Sequential Convolutional Neural Network** built with Keras.

```
Input: (128, 128, 3)  ← 128×128 RGB spectrogram image
        │
        ▼
Conv2D(32 filters, 3×3, ReLU)
        │
        ▼
MaxPooling2D(2×2)         ← Reduces spatial size by half
        │
        ▼
Conv2D(64 filters, 3×3, ReLU)
        │
        ▼
MaxPooling2D(2×2)         ← Reduces spatial size again
        │
        ▼
Flatten()                 ← Convert 2D feature maps to 1D vector
        │
        ▼
Dense(128, ReLU)          ← Fully connected layer, learns abstract features
        │
        ▼
Dropout(0.3)              ← Randomly drops 30% of neurons during training
        │
        ▼
Dense(11, Softmax)        ← Output probabilities for 11 genres
        │
        ▼
Output: genre label
```

### Layer-by-Layer Explanation

| Layer | Parameters | Purpose |
|---|---|---|
| `Conv2D(32, 3×3, relu)` | 32 filters × (3×3×3 + 1) = 896 | Detect low-level features: edges, textures |
| `MaxPooling2D(2,2)` | None | Down-sample, reduce computation, add translation invariance |
| `Conv2D(64, 3×3, relu)` | 64 filters × (3×3×32 + 1) = 18,496 | Detect higher-level patterns: shapes, motifs |
| `MaxPooling2D(2,2)` | None | Further down-sampling |
| `Flatten()` | None | Convert feature maps to a flat vector |
| `Dense(128, relu)` | varies | Learn abstract genre-discriminating features |
| `Dropout(0.3)` | None | Regularization — prevents overfitting |
| `Dense(11, softmax)` | 128×11 + 11 = 1,419 | Output a probability distribution over 11 genres |

### Compiler Settings

```python
model.compile(
    optimizer='adam',               # Adaptive learning rate optimizer
    loss='categorical_crossentropy', # Standard loss for multi-class classification
    metrics=['accuracy']
)
```

- **Adam optimizer** adapts the learning rate per parameter, making training faster and more stable.
- **Categorical cross-entropy** measures the difference between the predicted probability distribution and the true one-hot label.

## 9. Prediction Pipeline

```python
def predict_genre(audio_file):
    temp_path = 'temp_spec.png'
    save_spectrogram(audio_file, temp_path)         # Step 1: audio → image
    img = image.load_img(temp_path, target_size=img_size)
    img_array = image.img_to_array(img) / 255.0     # Step 2: normalize
    img_array = np.expand_dims(img_array, axis=0)   # Step 3: add batch dim
    prediction = model.predict(img_array)            # Step 4: get probabilities
    predicted_index = np.argmax(prediction)          # Step 5: top class
    predicted_genre = list(train_data.class_indices.keys())[predicted_index]
    os.remove(temp_path)                             # Step 6: cleanup
    return predicted_genre
```

**How it works:**
1. The audio file is converted to a mel spectrogram image and saved temporarily.
2. The image is loaded, resized to 128×128, and its pixels normalized to [0, 1].
3. A batch dimension is added (`expand_dims`) because Keras expects input as `(batch, height, width, channels)`.
4. `model.predict()` returns an array of 11 probabilities.
5. `np.argmax()` picks the index with the highest probability.
6. That index maps back to a genre name using `train_data.class_indices`.
7. The temporary spectrogram file is deleted.

## 11. Results & Evaluation

The model is evaluated using `model.evaluate(val_data)`, which outputs:
- **Loss** (categorical cross-entropy)

### Typical Results on GTZAN

| Metric | Approximate Value |
|---|---|

### Why the gap between train and val accuracy?
- The dataset is small (~1,000 clips), so the model can memorize training samples (overfitting).

## 12. Limitations & Future Improvements

### Current Limitations
| Limitation | Details |
|---|---|
| Small dataset | GTZAN has only 100 clips per genre, which is limited for deep learning |
| Fixed audio length | All clips are 30 seconds; real-world audio varies |
| No data augmentation | Training set could benefit from pitch shifting, time stretching, noise addition |
| Simple CNN | Only 2 conv layers; deeper architectures (ResNet, VGG) could improve accuracy |
| No model persistence | The trained model is not saved to disk (`model.save()`) |
| Single genre output | Predicts only one genre; real music often blends genres |

### Future Improvements
- **Save and load model** using `model.save('genre_classifier.h5')`
- **Data augmentation** on spectrograms (random flips, brightness shifts) or on raw audio (pitch shift, time stretch with Librosa)
- **Transfer Learning** using pre-trained image models (VGG16, EfficientNet) fine-tuned on spectrograms
- **Use MFCCs** (Mel-Frequency Cepstral Coefficients) alongside or instead of spectrograms
- **Add confusion matrix** to identify which genres are most often confused
- **Deploy as a web app** using Flask/FastAPI + a React frontend
- **Use CNNs + RNNs (CRNN)** to capture both spatial (CNN) and temporal (LSTM/GRU) features

## 14. Interview-Ready Version

> This section gives you crisp, clear, confident answers to questions an interviewer might ask about this project.

---

### Q1. Describe your project in one sentence.

**A:** I built a music genre classification system that converts raw audio files into mel spectrogram images and then uses a Convolutional Neural Network to classify them into one of 11 music genres with no manual feature engineering.

---

### Q2. Why did you convert audio to images? Why not feed raw audio to the model?

**A:** Raw audio is a 1D waveform with up to 660,000 data points for a 30-second clip at 22 kHz. Training a model on raw waveforms is computationally expensive and requires the model to learn temporal patterns from scratch. A mel spectrogram converts this into a 2D time-frequency image, where the model can reuse powerful **CNN architectures** already proven on image classification. Different genres produce visually distinct spectrogram patterns — for example, metal has dense high-frequency energy while classical is smoother — making the task well-suited for CNNs.

---

### Q3. What is a mel spectrogram? Why mel and not a regular spectrogram?

**A:** A spectrogram shows how frequency content changes over time. A **mel spectrogram** applies the **mel scale**, which is a perceptual scale of pitches designed to match human hearing. Human ears can distinguish small frequency differences at low frequencies but not at high frequencies. The mel scale compresses the high-frequency range and expands the low-frequency range to reflect this. This makes the spectrogram more perceptually meaningful and focuses the model's learning on what humans actually hear, which is more relevant to genre perception.

---

### Q4. Walk me through your CNN architecture.

**A:** My CNN has four main stages:

1. **Feature extraction — Conv Layer 1:** A `Conv2D` layer with 32 filters and 3×3 kernels + ReLU activation learns low-level visual features like edges and texture patterns in the spectrogram.
2. **Spatial reduction — MaxPool 1:** A `MaxPooling2D(2,2)` layer halves the spatial dimensions, reducing computation and adding translation invariance.
3. **Higher-level features — Conv Layer 2:** A `Conv2D` with 64 filters learns more abstract patterns like genre-specific spectral shapes.
4. **Classification — Dense Head:** After flattening, a `Dense(128)` layer learns genre-discriminating combinations of features, followed by `Dropout(0.3)` to prevent overfitting, and a final `Dense(11, softmax)` that outputs probabilities for each genre.

---

### Q5. Why did you use Dropout? What does it do?

**A:** Dropout is a regularization technique. During training, it randomly sets 30% of the neurons in the Dense layer to zero on each forward pass. This forces the network to not rely on any single neuron and learn redundant representations — making it more robust and less prone to overfitting. At inference time, Dropout is automatically disabled and all neurons contribute.

---

### Q6. Why did you choose categorical cross-entropy as the loss function?

**A:** Because this is a **multi-class classification** problem with mutually exclusive classes. Categorical cross-entropy measures the difference between the model's predicted probability distribution (from softmax) and the true one-hot encoded label. It heavily penalizes confident wrong predictions, encouraging the model to put probability mass on the correct class.

---

### Q7. Why Adam optimizer?

**A:** Adam (Adaptive Moment Estimation) is the go-to optimizer for deep learning because it combines the benefits of two other methods:
- **Momentum** (smooths gradients over time)
- **RMSProp** (adapts learning rate per parameter)

This means it converges faster than plain SGD and requires less tuning of the learning rate.

---

### Q8. How did you handle train/validation split?

**A:** I used Keras's `ImageDataGenerator` with `validation_split=0.2`, which automatically reserves 20% of the images for validation during training. This gives me an unbiased estimate of how the model generalizes to unseen data. I used the same generator for both splits to ensure consistent preprocessing (normalization by `1./255`).

---

### Q9. What accuracy did your model achieve?

**A:** On the GTZAN dataset with this architecture and 20 epochs of training, the model typically achieves around 70–80% validation accuracy. Training accuracy is higher (85–95%), which indicates some degree of overfitting due to the small dataset size (only 100 clips per genre). Adding data augmentation or using transfer learning could push validation accuracy higher.

---

### Q10. What are the limitations of your project?

**A:** Three main limitations:
1. **Small dataset:** GTZAN has only 1,000 total clips; modern deep learning performs better with tens of thousands of samples.
2. **Simple architecture:** Only 2 conv layers. Deeper models like ResNet or EfficientNet pre-trained on ImageNet could leverage more powerful features.
3. **No model persistence:** The trained model isn't saved; every session re-trains from scratch. Adding `model.save('model.h5')` and `load_model()` would fix this.

---

### Q11. How would you improve this project?

**A:** Several ways:
- **Data augmentation** on spectrograms (random cropping, brightness changes) or on raw audio (pitch shifting, time stretching with Librosa) to artificially expand the dataset.
- **Transfer learning** with VGG16 or EfficientNetB0 pretrained on ImageNet — spectrograms look like images, so these features transfer reasonably well.
- **CRNNs** (Convolutional Recurrent Neural Networks): use CNN to extract spatial features and then an LSTM/GRU to model temporal dynamics across the spectrogram.
- **Deploy it**: wrap the model in a Flask API, build a simple frontend where users upload a `.wav` file and get the genre back in real time.

---

### Q12. What is the difference between `model.fit()` and `model.evaluate()`?

**A:** `model.fit()` is used during **training** — it adjusts the model's weights using backpropagation over multiple epochs. `model.evaluate()` is used **after training** to compute the loss and accuracy on a fixed dataset (in this case, the validation set) without updating any weights. It gives an unbiased performance estimate.

---

### Q13. Why did you use `np.expand_dims()` during prediction?

**A:** Keras expects input in the shape `(batch_size, height, width, channels)`. When predicting a single image, the loaded image has shape `(128, 128, 3)`. `np.expand_dims(img_array, axis=0)` adds a batch dimension at position 0, changing the shape to `(1, 128, 128, 3)` — meaning "a batch of one image."

---

### Q14. If asked to scale this to millions of songs, what changes would you make?

**A:** Several architectural changes:
- **Pre-compute spectrograms** once and store them, rather than computing on the fly.
- **Use TFRecord format** for efficient TensorFlow data pipelines with `tf.data`.
- **Larger, pretrained models** (EfficientNet, AST — Audio Spectrogram Transformer) for better accuracy.
- **Distributed training** using `tf.distribute.MirroredStrategy` across multiple GPUs.
- **Model serving** with TensorFlow Serving or TorchServe behind a load balancer.
- **Database-backed metadata** to store predicted genres alongside track IDs.
